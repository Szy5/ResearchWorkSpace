## 一、背景与目标

### 1.1 为什么需要独立的管理页面

### 1.2 范围确认

本文档覆盖以下五个模块（按用户确认的范围）：

1. 论文管理（批量与危险操作，区别于 `Dashboard` 的浏览场景）
2. 任务历史与知识图谱数据管理

### 1.3 非目标

- 多用户、权限体系（RBAC）——现状与 v1 Web 一致，假设本地单用户，鉴权留给"上线到多用户前必须补上"的既有待办
- 知识图谱可视化探索（力导向图、拖拽交互）——那是 `GraphViewer`（需求文档 10.3 流程4）的职责，管理页面里的图谱模块只做**数据纠错**，不做探索式可视化
- Prompt 在线执行/调试（例如页面里直接跑一次 LLM 预览新 prompt 的效果并写入正式产物）——涉及成本和幂等性，放到后续版本单独评估

## 二、信息架构

管理页面作为顶部导航新增的一级入口 `⚙️ 管理`（区别于第十节草图里泛泛的 `Settings`，管理页面就是 `Settings` 的正式落地和扩展），下设五个 Tab：

```
⚙️ 管理
├── 📄 论文管理        批量删除/批量标记/孤儿检测/存储占用
└── 🗂️ 任务与图谱       任务历史、失败重试、图谱节点/关系纠错
```

## 三、模块详细设计

### 3.1 论文管理

**现状**：`Dashboard`（`GET /api/papers`）已经能浏览+过滤+进入详情编辑单篇论文，但缺少批量与"危险操作"：

- 没有删除入口——`raw/{slug}/` 和 `artifacts/{slug}/` 只能手动 `rm -rf`
- 没有批量标记 `reviewed`/`tags`

**用户故事**：

> 作为使用者，我攒了三个月的论文后想清理掉当初手滑加进来、读了发现不相关的几十篇论文，我希望能勾选批量删除，而不是一篇篇进详情页操作，也不用登录服务器敲命令。

**功能清单**：

| 功能                                              | 优先级 | 说明                                                                                                                                                                                |
| ------------------------------------------------- | ------ | ----------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| 论文列表（复用现有`GET /api/papers`，增加多选） | P0     | 支持按`reviewed`/`tags`/`primary_pattern`/`year`/`venue` 组合过滤，表格视图（区别于 `Dashboard` 的卡片视图，管理场景信息密度优先）                                      |
| 批量标记`reviewed`                              | P0     | 复用`PATCH /api/papers/{slug}/meta`，管理页面循环调用或新增 `POST /api/papers/batch-review` 减少请求数                                                                          |
| 单篇/批量删除                                     | P0     | 级联删除`raw/{slug}/`、`artifacts/{slug}/`；如该论文已 `reviewed` 并进入图谱，需同时提示"是否一并移除图谱节点及相关前作关系"（详见 3.5）；必须二次确认（不可逆操作）          |
| 孤儿检测                                          | P1     | 扫描`raw/*` 与 `artifacts/*` 目录名差集，列出"未摄入"（有 raw 无 artifacts）和"源文件缺失"（有 artifacts 无 raw）两类，各自给出对应操作入口（一键 ingest / 标记为"仅保留产物"） |
|                                                   |        |                                                                                                                                                                                     |

**新增接口**：

```
DELETE /api/papers/{slug}              # 级联删除 raw + artifacts（+ 可选图谱节点）
POST   /api/papers/batch-delete        # body: {slugs: [...], purge_graph: bool}
POST   /api/papers/batch-review        # body: {slugs: [...], reviewed: bool}
GET    /api/papers/orphans             # 孤儿检测结果
```

**风险提示**：删除操作目前没有任何撤销机制（不像 `manifest.updated_at` 乐观锁那样"失败重试即可"，这是真实的数据丢失）。建议 P0 阶段就要求二次确认弹窗展示"将删除的文件路径列表"，而不是只显示论文标题；

### 3.2 任务历史与知识图谱数据管理

**现状拆成两块**：

**（a）任务历史**：`web/services/job_manager.py: JobManager` 是纯内存 `dict[str, JobRecord]` + 线程池，v1 设计里已明确写了"不做崩溃恢复持久化队列，进程重启后运行中任务丢失"（技术方案 11.3 节）。这意味着即使当前请求成功了，一旦进程重启，"过去跑过哪些 job、成功率如何"的记录也一并消失，用户排查"昨天那次批量生成为什么有 3 篇失败了"无据可查。

**（b）知识图谱数据**：`graph/state_store.py: GraphStateStore` 维护 `graph_state/papers.json` + `graph_state/prior_work_relations.json` 快照 + `graph_updates/graph_updates.jsonl` 事件日志。节点包含 `is_stub`（外部占位节点，尚未完整收录）标记；关系来自 LLM 抽取的 `prior_works.json`，存在幻觉风险（技术方案 3.5 节已提到"LLM 提取前作时存在幻觉风险"）。目前只能手改 JSON 文件来修正错误的节点/关系，且改完还要重新走 `paper-wiki graph apply` 才能同步到 Neo4j。

**用户故事**：

> 作为使用者，我批量生成了 10 篇论文，3 篇失败了，我想知道是哪 3 篇、失败原因是什么，方便的话直接点"重试"，而不是回去翻终端日志；另外我发现某篇论文的 Prior Works 里 LLM 把两篇不同论文识别成了同一个节点（标题打错了导致没匹配上已收录论文，生成了多余的 stub 节点），我想在页面上把这个 stub 节点和正确的论文合并，而不是手改 `graph_state/papers.json`。

**功能清单**：

| 功能                                    | 优先级         | 说明                                                                                                                                                                                                                                                                                      |
| --------------------------------------- | -------------- | ----------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| 任务历史持久化                          | P0（前置依赖） | `JobManager` 增加落盘：任务终态（`succeeded`/`failed`）写入 `artifacts/.jobs/{date}.jsonl` 或类似结构，内存态仍用于当前进程的实时轮询，落盘态用于历史查询——这是"轻量持久化"而非引入 Celery/Redis，与技术方案 11.8 节"明确不在 v1 引入 Celery/Redis"的既有取舍保持一致           |
| 任务历史列表                            | P0             | 按 slug/target/status/时间筛选，展示失败任务的`error` 信息                                                                                                                                                                                                                              |
| 失败任务重试                            | P1             | 复用现有`POST /api/papers/{slug}/ingest` 等触发接口，管理页面只是"用历史记录里的参数再触发一次"的快捷入口，不是新的执行路径                                                                                                                                                             |
| 图谱节点/关系浏览（表格视图，非可视化） | P1             | 列出`graph_state/papers.json` 中的节点（区分 `is_stub` 与已收录），列出关系及其 `role`/`relationship_sentence`；这里刻意用表格而非力导向图——探索式可视化是 `GraphViewer` 的职责（见 1.3 非目标），管理页面要解决的是"精确定位到某条脏数据并修正"，表格+搜索比图更适合这个场景 |
| 手动修正关系                            | P1             | 编辑/删除某条`PriorWorkRelation`（错误的 role 或幻觉产生的关系），保存后需要生成对应的修正事件追加到 `graph_updates.jsonl`（保持现有"事件日志可幂等重放"的设计不变，不能绕过事件机制直接改快照文件）                                                                                  |
| Stub 节点合并                           | P2             | 将一个`is_stub=true` 的外部节点与后续被正式收录的论文节点合并（合并后该节点所有入边/出边的 `target`/`source` 需要重新指向真实 slug），这是本模块里最复杂的操作，涉及事件日志的正确性，建议单独设计一次评审再实现                                                                    |

**新增接口**：

```
GET   /api/jobs/history                    # ?slug=&status=&since= 过滤，读落盘历史
GET   /api/graph/nodes                     # 分页浏览节点，?is_stub= 过滤
GET   /api/graph/relations                 # 分页浏览关系
PATCH /api/graph/relations/{relation_key}  # 修正 role / relationship_sentence，追加修正事件
DELETE /api/graph/relations/{relation_key} # 删除关系，追加删除事件
POST  /api/graph/nodes/{stub_id}/merge     # body: {into_slug}，合并 stub 节点（P2）
```

## 四、跨模块设计原则

延续现有 Web v1 的既定原则（技术方案 11.0 节），管理页面额外强调：

1. **管理页面不新增数据源**：所有数据仍然来自 `artifacts/`、`graph_state/`、`.env`、`published/` 等既有文件系统位置；新增的 `publish_log.jsonl`、`.jobs/{date}.jsonl` 是对现有"文件系统即事实来源"原则的延伸，不是引入数据库
2. **危险操作不可逆时必须二次确认**：删除论文、撤下发布、合并图谱节点，这三类操作都会造成数据丢失或不可逆变更，UI 上必须有明确的二次确认，且确认弹窗要展示"具体将发生什么"（文件路径/记录数），不能只是通用的"确定要删除吗？"
3. **图谱数据修正必须走事件日志，不允许直接改快照**：这是现有 `graph/` 包"事件日志支持幂等重放"设计的硬约束（技术方案 7.3 节），管理页面对图谱的任何写操作都必须落一条 `GraphEvent`，而不是直接 `save_state()` 覆盖快照，否则会破坏"从事件日志可以重建快照"的不变式
4. **Prompt 编辑能力暂缓，等模板与代码解耦后再开放**：见 3.2 节的安全风险分析，这是本文档里唯一一个"能力上限"层面的克制

## 五、优先级与里程碑建议

| 阶段                   | 交付内容                                                                                                | 说明                                                                                       |
| ---------------------- | ------------------------------------------------------------------------------------------------------- | ------------------------------------------------------------------------------------------ |
| P0（最小可用管理页面） | 论文批量删除/标记、Prompt 只读浏览+默认版本选择、系统配置查看+编辑（不含连接测试）、发布记录持久化+列表 | 覆盖"最常见的手动操作需求"，其中发布记录持久化和任务历史持久化是后续模块的地基，需要优先做 |
| P1                     | 孤儿检测、配置连接测试、发布撤下/重新发布、任务历史列表+重试、图谱节点/关系表格浏览+手动修正            | 补齐"排查问题"和"数据纠错"能力                                                             |
| P2                     | 存储占用统计、Prompt 版本 diff、多平台发布统计、Stub 节点合并                                           | 锦上添花，或者复杂度较高需要单独评审的功能                                                 |

## 六、待决策事项

| 问题                          | 选项 A                                          | 选项 B                                         | 建议                                                                                    |
| ----------------------------- | ----------------------------------------------- | ---------------------------------------------- | --------------------------------------------------------------------------------------- |
| 论文删除是否需要软删除/回收站 | 直接物理删除（二次确认兜底）                    | 移入`.trash/` 保留 N 天                      | 先选 A，本地工具磁盘可控；如果误删事故发生过一次，再升级到 B                            |
| Prompt 默认版本配置粒度       | 全局默认（`Settings` 新增字段，所有论文共用） | 按论文/按 tag 设置不同默认                     | 先选 A，按论文级配置在还没有真实需求前是过度设计                                        |
| 配置修改后是否自动生效        | 保存即清空`get_settings()` 缓存，立即生效     | 保存后提示需重启服务                           | 建议 A，见 3.3 节，否则管理页面的"配置"价值有限                                         |
| 图谱数据修正的落地时机        | 和管理页面一起做（P1）                          | 独立评审排期（涉及事件日志正确性，复杂度较高） | 建议先把 P0 范围（论文/Prompt/配置/发布记录）跑通上线，图谱修正单独开一次设计评审再排期 |
