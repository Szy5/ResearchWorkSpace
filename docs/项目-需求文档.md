> 版本：v2.0 | 创建：2026-06-23 | 最后更新：2026-07-14 | 状态：持续维护
> **定位转变**：从学术知识库系统 → 科研日常工作站

## 一、产品定位与核心价值

### 用户场景

你每天来到工位，打开 **Paper-Wiki 工作站**，论文有两种方式进入你的视野：

- 🌅 **每日推荐**（Paper2Recommend）：系统基于你读过/收藏过的论文口味，从当天arXiv新增里挑出最相关的一批，打开工作站就能看到，不用主动去刷
- 🔎 **主动检索 / 按ID添加**（Paper2Search）：临时想读某方向的论文就搜关键词；已经知道某篇论文的arXiv ID，粘贴进去系统自动拉取PDF/LaTeX源文件

不管论文从哪条路线进来，后面都走同一条主线：

- ✅ **批量筛选**：从推荐列表/检索结果里勾选自己认为重要的论文——不是一篇篇点开决定，而是先扫一遍、批量勾选
- ⚙️ **批量生成**：勾选后一键触发，后台批量为选中论文生成 Summary、Prior Works、科学范式分类
- 📖 **阅读与审查**：生成完成后逐篇打开精读，边读边修正LLM可能出错的信息
- ✔️ **确认完成**：修正无误后点击 **Reviewed**，这篇论文正式入库，工作流闭环
- ✍️ **按需产出Blog**：对想对外分享的论文，一键转换为可发布Blog（微信公众号、Markdown等）
- 💡 **知识积累**：所有Reviewed论文自动连接，形成知识图谱，支持后续IDEA发现

**核心目标**：让"发现论文"这件事不再需要主动搜、主动刷，系统每天主动喂给你最值得读的一批；剩下的精读、审查、传播、积累，在同一个工作站里一站式完成。

### 区别于传统文献管理

| 维度     | 传统做法                    | Paper-Wiki                             |
| -------- | --------------------------- | -------------------------------------- |
| 论文发现 | 主动刷arXiv/Twitter，容易漏 | 每日推荐主动推送 + 检索/按ID添加双通道 |
| 阅读效率 | 手工摘要，易遗忘            | 批量自动生成Summary，多次复习          |
| 内容输出 | 论文→个人笔记→不外传      | 论文→Blog→微信公众号→读者           |
| 知识关联 | 孤立论文本                  | 自动构建论文关系网，发现IDEA           |
| 研究方向 | 感觉模糊                    | 知识图谱展示学科演化脉络               |

## 二、核心工作流

```
┌───────────────────────────────────────────────────────────────────────┐
│                         Paper-Wiki 工作站                              │
├───────────────────────────────────────────────────────────────────────┤
│  发现（两条路线，二选一或都用）                                        │
│   🌅 Paper2Recommend          🔎 Paper2Search                          │
│   每日候选池+Zotero口味       NL检索 / arXiv ID直接拉取                │
│           └──────────────┬──────────────┘                             │
│                           ▼                                            │
│                  ✅ 批量筛选（勾选感兴趣的论文）                       │
│                           ▼                                            │
│                  ⚙️ 批量理解生成（Summary/Prior Works/Pattern）        │
│                           ▼                                            │
│                  📖 阅读 + 人工审查（边读边修正）                      │
│                           ▼                                            │
│                  ✔️ 点击 Reviewed（闭环完成，入知识库）                │
│                     ↓                              ↓                   │
│           ✍️ 按需生成Blog并发布              📊 汇入知识图谱          │
└───────────────────────────────────────────────────────────────────────┘
```

### 2.1 论文发现：两条路线

#### 路线A：每日推荐（Paper2Recommend驱动）

**参考实现**：VibeIDEA `backend/recommender.py` + `backend/main.py` 的 Zotero语料 + arXiv候选 + embedding重排序方案，算法思路直接复用。

**每日自动运行**（定时任务，用户不用手动触发）：

1. 拉取"口味语料"：读取用户Zotero库中的论文摘要（`abstractNote`），按加入时间做时间衰减加权（越新收藏权重越高）
2. 拉取候选池：优先按分类订阅arXiv每日新增 RSS（如 `cs.AI+cs.LG+cs.CL+cs.IR`），RSS 不可用时 fallback 到 arXiv export API 的 `submittedDate` 查询
3. 相似度重排序：用sentence-transformers计算候选论文摘要与口味语料的余弦相似度，乘以时间衰减权重，取Top K（默认15篇）
4. 落盘：候选池快照写入 `artifacts/.recommendations/{date}.json`，用户打开工作站首页直接看到

**当前实现状态（2026-07-14）**：CLI `paper-wiki recommend run` 已落地。真实端到端验证 `--max-papers 3` 成功，`corpus_size=192`、`candidate_pool_size=200`、`degraded=False`，写入 `artifacts/.recommendations/2026-07-14.json` 和 `latest.json`。Zotero ID 兼容纯数字 ID，也兼容 `users/...` / `groups/...` URL 片段。

**用户操作**：

```bash
# 手动触发一次（也可配置每日定时任务自动跑）
paper-wiki recommend run --max-papers 15
```

#### 路线B：主动检索 / 按ID添加（Paper2Search驱动）

**参考实现**：Microsoft ResearchStudio 的 `paper_search` skill —— 按主题检索走arXiv官方API，遵守限流礼仪（跨进程节流，请求间隔≥4秒）；按ID拉取源码沿用VibeIDEA `paper.py` 下载tarball、按`.bbl`匹配主tex文件的逻辑。

**子能力1：按主题检索**（找相关工作/前沿论文）

```bash
paper-wiki search "transformer attention mechanism" --start-year 2023
```

- v1已接入arXiv export API（与现有Paper2Assets摄入管道天然契合），检索结果统一为 `title/authors/year/abstract/url/venue/arxiv_id` schema，与 `core/models.py: PaperMeta`、`assets/models.py: ReferenceEntry` 字段对齐，避免重复定义转换逻辑
- v2再扩展多源并发（Semantic Scholar/OpenAlex/Crossref等），参考ResearchStudio的多线程并发检索架构；"model_knowledge"式LLM兜底召回有幻觉风险，需评估后再决定是否引入

**子能力2：按arXiv ID直接添加**（已知论文，直接拉取）

```bash
paper-wiki fetch 2401.12345            # 只拉取PDF/LaTeX到 raw/{slug}/
paper-wiki fetch 2401.12345 --and-ingest   # 拉取后立即生成 summary/prior_works/sci_pattern
```

当前 `fetch` 会先下载源码/PDF 到临时目录，校验能识别主 `.tex` 后再落盘到 `raw/{slug}/`，避免半成品污染 Layer 0。真实验证中 `paper-wiki fetch 2406.00552` 成功，随后 `parse` 可通过。

无论走哪条路线，产出都是"候选论文列表"（推荐路线是自动产出的当日快照，检索路线是即时查询结果），下一步都是**批量筛选**。

### 2.2 批量筛选与批量理解生成（MVP）

**用户操作**（Web界面上勾选多篇 + 一次性触发；CLI上传入多个slug）：

```bash
# CLI：批量为多篇论文生成三件套
paper-wiki ingest slug-1 slug-2 slug-3

# 或者：扫描raw/下所有待处理论文，自动补齐
paper-wiki ingest-all
```

**系统生成产物**（每篇论文排队/并发处理，自动完成，用户不感知）：

1. **Paper Assets**（deterministic，不依赖LLM）

- `manifest.json`：论文元数据、文件清单
- `paper.md`：规范化论文正文
- `sections.json`：章节树和正文
- `figures/`：提取的图片
- `references.json`：文献列表

2. **独立语义产物**（基于LLM，可分别生成/重跑）

- `summary.md`：精读笔记（按"背景→方法→结果→启发"结构）
- `prior_works.json`：前作分析（直接影响本论文的5-7篇论文+关系）
- `sci_pattern.json`：创新范式分类（用15类科学模式标记）

**检查点**：批量生成完成后，这些论文进入"待审查"队列。

### 2.3 阅读与人工审查（MVP）

这是整条工作流真正的"精读"环节——批量生成解决的是"值得读的论文都有结构化材料了"，人工审查解决的是"确保材料是对的"。

**用户操作**：

1. 打开"待审查"列表，逐篇进入详情页
2. 阅读自动生成的 Summary / Prior Works / Pattern
3. 发现LLM幻觉或元数据错误（标题、作者、arXiv ID等）时直接在页面上修正
4. 确认无误后点击 **Reviewed**，`manifest.paper.reviewed` 置为 `true`

**闭环完成**：只有 `reviewed: true` 的论文才会汇入知识图谱、才能被发布为Blog。

### 2.4 Blog生成与校验发布（MVP，按需触发）

不是每篇Reviewed论文都要发Blog，这是一个独立的、按需触发的动作：

```bash
# 查看自动生成的Blog预览
paper-wiki preview <slug>

# 人工修改后发布到微信公众号
paper-wiki publish wechat <slug> \
  --html artifacts/slug/blog.html \
  --cover path/to/cover.jpg
```

**Blog生成逻辑**：

- 输入：`summary.md` + 论文assets
- 输出：HTML格式Blog（支持微信公众号排版）
- 内容：开篇吸引 → 背景问题 → 核心方法 → 实验结果 → 个人评注

**发布流程**：

1. 人工审查Blog内容（修复LLM幻觉、调整格式）
2. 选择封面图
3. 一键发布到微信公众号（创建为草稿，保留人工最终确认）

### 2.5 知识积累与IDEA发现（后续）

**系统自动构建知识图谱**：

- 节点：已审查（`reviewed: true`）的论文
- 关系：来自 `prior_works.json` 的直接思想传承
- 属性：论文年份、科学范式、关键概念

**支持的查询**：

- 📍 **溯源查询**：这篇论文在谁的肩膀上？沿思想链往上追
- 🎯 **影响力查询**：哪些后续工作基于我读过的这篇论文？
- 💡 **范式聚类**：这个方向用过哪些科学范式？有什么范式空白？
- 🔗 **关联路径**：两篇论文之间是否有思想传承链？

**IDEA发现应用**：

- 发现范式空白 → 新研究方向
- 识别跨域综合机会（不同领域的论文在这里碰撞）
- 追踪最前沿工作（图谱边界节点最新的论文）

## 三、系统架构（简化版）

不再用Layer 0-3，改为五个核心模块，统一用 `Paper2X` 命名：

```
Paper-Wiki = 【Paper2Search + Paper2Recommend】 + Paper2Assets + Paper2Insight + Paper2Blog + Paper2Graph
              └──────── 发现层 ────────┘         摄入管道       理解引擎       发布接口      知识引擎
```

### 3.1 发现层（Discovery Layer）—— 新增

对应工作流2.1的两条路线，负责"论文从哪来"，产出统一的候选论文列表，交给 Paper2Assets 处理。**发现层只负责"找到论文+落到raw/或候选快照"，不调用 Paper2Insight 的LLM**——批量筛选之后才会触发摄入。

**关键设计**：Paper2Search 是发现层里唯一直连外部数据源的模块；Paper2Recommend **不重新实现检索**，只是调用 Paper2Search 拿候选池 + 拿Zotero语料，再做重排序。这样"拉取arXiv候选"这件事只有一份实现，Search和Recommend共享，不会出现两套HTTP客户端/限流逻辑各自维护的情况。

```
Paper2Search（检索适配器层，唯一直连外部数据源）
├── arxiv 源      —— 关键词检索 / 按ID拉取 / 每日新增RSS（失败时fallback到export API）
├── zotero 源     —— 读取用户Zotero库（这里Zotero是一个"语料源"，不是论文来源）
└── (v2) semantic_scholar / open_alex / crossref 源

Paper2Recommend（重排序层，调用Paper2Search，不直连外部API）
= Paper2Search.daily_candidates(ARXIV_QUERY)     # 候选池
  × 相似度(Paper2Search.zotero_corpus())          # 口味语料
  → Top K
```

#### Paper2Search（统一检索层）

> 📄 详细技术方案（模块结构、数据模型、限流设计、Mermaid流程图）：[Paper2Search 技术方案.md](<./Paper2Search%20技术方案.md>)

**定位**：所有"论文从哪查到"的能力收口在这一个模块，其余模块（包括Paper2Recommend）只消费它的输出。

**参考实现**：Microsoft ResearchStudio 的 `paper_search` skill（多源检索架构 + 限流礼仪）+ VibeIDEA `paper.py`（arXiv tarball下载与主tex文件匹配）。

**四个能力，共享同一套arXiv客户端/限流器**：

1. **按主题检索** `search(query, start_year, end_year) -> list[Candidate]`（用户直接可见）

   ```bash
   paper-wiki search "transformer attention mechanism" --start-year 2023
   ```

   v1只接入arXiv官方API，遵守限流礼仪（跨进程节流，请求间隔≥4秒，避免被封）；v2按需扩展多源并发（Semantic Scholar/OpenAlex/Crossref等），参考ResearchStudio的多线程并发检索架构；"model_knowledge"式LLM兜底召回有幻觉风险，需评估后再决定是否引入
2. **按ID直接拉取** `fetch(arxiv_id) -> raw/{slug}/`（用户直接可见）

   ```bash
   paper-wiki fetch 2401.12345 [--and-ingest]
   ```

   下载PDF+LaTeX源码tar包，解析主tex文件（参考VibeIDEA `paper.py` 按`.bbl`匹配主文件的逻辑），落盘到 `raw/{slug}/`
3. **每日候选池** `daily_candidates(category_query) -> list[Candidate]`（内部接口，供Paper2Recommend调用，不单独暴露CLI/API）

   - 拉取当日arXiv新增（`rss.arxiv.org/atom/{query}`），返回schema与 `search()` 完全一致
4. **Zotero语料源** `zotero_corpus(zotero_id, zotero_key) -> list[dict]`（内部接口，供Paper2Recommend调用作为"口味语料"；Zotero本身不产出待摄入论文，只是参考语料）

**统一Candidate schema**：`title/authors/year/abstract/url/venue/arxiv_id/source`，与 `core/models.py: PaperMeta`、`assets/models.py: ReferenceEntry` 字段对齐，避免在Web/CLI层重复定义转换逻辑。

#### Paper2Recommend（每日推荐，薄封装层）

> 📄 详细技术方案（相似度算法、优雅降级、语料源可插拔设计、Mermaid时序图）：[Paper2Recommend 技术方案.md](<./Paper2Recommend%20技术方案.md>)

**定位**：不直连任何外部API，只做"调用Paper2Search两次 + 重排序"。

**处理**（算法思路参考 VibeIDEA `backend/recommender.py` + `backend/main.py`）：

1. 调用 `Paper2Search.zotero_corpus()` 拉取口味语料，按收藏时间做时间衰减加权
2. 调用 `Paper2Search.daily_candidates(ARXIV_QUERY)` 拉取候选池
3. 相似度重排序：sentence-transformers embedding余弦相似度 × 时间衰减权重，取Top K（默认15篇）
4. 落盘：候选池快照写入 `artifacts/.recommendations/{date}.json`（快照而非最终产物，用户勾选后才转成正式摄入任务）

```bash
paper-wiki recommend run --max-papers 15
```

**v2方向**：语料源从外部Zotero切换/补充为Paper-Wiki自己 `manifest.paper.reviewed: true` 的论文集合，让推荐系统随着使用自我强化，逐步减少对外部Zotero账号的依赖——实现上只是给 `zotero_corpus()` 增加一个新的语料provider，Paper2Recommend本身的重排序逻辑不用改。

### 3.2 Paper2Assets（摄入管道 / Ingestion Pipeline）

**输入**：

- LaTeX 源文件或 PDF
- 用户提供的论文文件夹 `raw/{slug}/`

**处理**：

1. LaTeX 解析：递归内联 `\input`、提取章节、识别图片
2. 资产构建：生成deterministic assets（不依赖LLM，可重复）
3. 元数据提取：标题、作者、发表会议、arXiv ID等

**输出**：`artifacts/{slug}/assets/` + 独立语义产物文件

### 3.3 Paper2Insight（理解引擎 / Comprehension Engine）

**基于LLM的独立语义产物生成**：

这三个产物共享 `Paper Assets` 作为 canonical input，但不互相读取、不要求一次性绑定生成。CLI 默认会按顺序生成三者；需要单独重跑时，可通过 `--only summary|prior_works|sci_pattern` 选择目标。

1. **Summary Generator**

- 输入：Paper Assets
- 输出：`summary.md`
- 结构：由当前使用的 prompt 模板决定，不是系统硬编码的固定章节，详见5.2（当前默认 `paper_summary_v3.py` 是"研究动机与贡献→方法介绍→实验结果→简要总结"4节结构）
- 后处理：当 `prior_works.json` 或 `sci_pattern.json` 可用时，在文末追加 `## 科学发现范式` 和 `## 先前工作分析`
- Prompt版本：支持v1/v2/v3多版本，可自定义

2. **Prior Works Generator**

- 输入：Paper Assets
- 输出：`prior_works.json`（结构化JSON）
- 提取5-7篇核心前作，标注6种关系角色
- 包含200-300字综合叙述

3. **Pattern Classifier**

- 输入：Paper Assets
- 输出：`sci_pattern.json`（科学创新范式）
- 使用15类科学范式框架（来自Sci-Reasoning）
- 标注主要范式、置信度、分类理由

### 3.4 Paper2Blog（发布接口 / Publishing Interface）

**当前实现**：

- 微信公众号集成：生成草稿、上传图片、自动排版
- 本地HTML导出：用户可自定义样式后发布

**后续规划**：

- 多平台发布（知乎、Medium等）
- 视频/Reel自动生成（类ResearchStudio-Reel）

### 3.5 Paper2Graph（知识引擎 / Knowledge Engine）

**当前实现**：

- Neo4j图谱：存储论文和关系
- 增量更新：`graph_state/` 快照 + `graph_updates.jsonl` 事件日志

**后续规划**：

- 向量索引：embedding论文Summary和概念
- RAG检索：自然语言查询
- 图谱可视化：交互式展示论文关系网

## 四、数据组织结构

```
Paper-wiki/
├── raw/                     # 📥 输入：用户上传的原始论文
│   └── {paper-slug}/
│       ├── main.tex         # LaTeX主文件（推荐）
│       ├── paper.pdf        # PDF文件（可选）
│       └── ...
│
├── artifacts/               # 🔍 理解产物：Assets + 独立语义产物
│   ├── .recommendations/    # 📬 Paper2Recommend 每日候选快照（非最终产物，可随时重建）
│   │   ├── latest.json
│   │   └── 2026-07-14.json
│   └── {paper-slug}/
│       ├── manifest.json                # Assets入口与元数据
│       ├── assets/
│       │   ├── paper.md                 # 规范化论文主文
│       │   ├── sections.json            # 章节树
│       │   ├── figures/                 # 提取的图片
│       │   └── references.json          # 文献列表
│       ├── summary.md                   # 精读笔记
│       ├── prior_works.json             # 前作分析
│       ├── sci_pattern.json             # 范式分类
│       └── blog.html                    # Blog HTML（可选，用户生成）
│
├── published/               # ✅ 发布：已发布的Blog（HTML格式，非Markdown）
│   └── {platform}/
```

## 五、核心概念详解

### 5.1 Paper Assets（确定性资产）

**目的**：为论文提供稳定的中间层，避免下游重复解析LaTeX。

**核心特性**：

- ✅ Deterministic：同一份LaTeX输入，生成结果完全相同（无LLM调用）
- ✅ 可复用：多个下游流程（Blog生成、图谱构建、RAG索引）都依赖assets
- ✅ 版本化：`manifest.json`中记录parser版本、生成时间、源文件hash

**最小契约**包含：

| 文件                       | 作用                                             |
| -------------------------- | ------------------------------------------------ |
| `manifest.json`          | bundle入口，版本、路径、计数、warnings           |
| `assets/paper.md`        | 规范化论文正文，保留标题、章节、公式、图片占位符 |
| `assets/sections.json`   | 结构化章节树，包含source span和citations         |
| `assets/figures/`        | 提取的图片，附带caption和label                   |
| `assets/references.json` | 文献候选列表                                     |

---

### 5.2 Summary.md（精读笔记）

**作用**：将论文压缩为一份精炼、图文并茂、面向广泛学术读者的研究博客风格Markdown（"帮读者在五分钟内理解论文关键思想、方法、核心贡献"）。

**结构由 prompt 模板决定，不是系统硬编码的固定章节**——这是 Prompt 多版本机制（见7.1）的直接体现：换一个 `--summary-prompt` 就能得到不同的章节结构，`summary_generator.py` 不对章节名做任何假设，只负责清洗输出（剥离代码块包裹、剥离frontmatter）。

**当前默认版本 `paper_summary_v3.py` 的结构**（CLI `--summary-prompt` 默认值）：

```
#  {论文标题}
## 1. 研究动机与贡献
## 2. 方法介绍
## 3. 实验结果
## 4. 简要总结
```

**`paper_summary_v2.py` 的结构**（更详细，8节）：

```
#  {论文标题}
## 1. 研究动机
## 2. 核心洞见
## 3. 提出的方法（含子章节）
## 4. 实验结果
## 5. 分析与洞察
## 6. 主要贡献
## 7. 局限性与未来工作
## 8. 总结
```

**重要边界**：`summary.md` 本身**不包含 frontmatter**——`SummaryGenerator._strip_existing_frontmatter()` 会主动剥离模型可能生成的 YAML frontmatter 或伪frontmatter，因为论文元信息（标题、作者、贡献类型等）统一维护在 `manifest.json` 的 `paper` 字段里，不允许在 summary.md 里重复维护一份、造成两处不一致。

**贡献类型**（`manifest.json: paper.contribution_type`，人工审查时填写/修正，非LLM在summary.md里自行标记）：

- 问题定义型：重新定义了失败场景或研究问题
- 机制解释型：揭示了模型/系统为什么表现出某种行为
- 方法改进型：提出了更好的新方法
- 评测基准型：构建了数据集、基准或评测框架

---

### 5.3 Prior Works（前作分析）

**作用**：捕捉论文的"学术DNA"——它直接站在哪些工作的肩膀上。

**包含内容**：

- 5-7篇直接影响论文核心创新的前作
- 每篇前作的标题、作者、年份、arXiv ID
- 关系角色：Baseline / Inspiration / Gap Identification / Foundation / Extension / Related Problem
- 与当前论文的直接关系（一句话）

**综合叙述**：200-300字说明这些前作如何共同启发了当前工作

**作用**：直接用于构建知识图谱的边

---

### 5.4 Sci-Pattern（科学范式分类）

**作用**：用标准化模式标记论文的创新行为，支持跨论文对比和IDEA发现。

**15类科学创新范式**（来自Sci-Reasoning）：

| ID  | 范式                               | 核心描述                         |
| --- | ---------------------------------- | -------------------------------- |
| P01 | Gap-Driven Reframing               | 发现具体局限，重新定义问题       |
| P02 | Cross-Domain Synthesis             | 从其他领域引入思想或机制         |
| P03 | Representation Shift               | 替换核心表示，简化问题结构       |
| P04 | Modular Pipeline Composition       | 分解为可组合的专用模块           |
| P05 | Data & Evaluation Engineering      | 构建数据集、基准或评测框架       |
| P06 | Principled Probabilistic Modeling  | 以概率模型替换启发式方法         |
| P07 | Formal-Experimental Tightening     | 实验观察→理论形式化→验证的闭环 |
| P08 | Approximation Engineering          | 设计保持理论性质的可扩展近似     |
| P09 | Inference-Time Control             | 推理时控制代替重训练             |
| P10 | Inject Structural Inductive Bias   | 将领域结构注入模型               |
| P11 | Multiscale & Hierarchical Modeling | 粗到细的层次建模                 |
| P12 | Mechanistic Decomposition          | 分解可解释机制并验证             |
| P13 | Adversary Modeling                 | 显式建模对手行为                 |
| P14 | Numerics & Systems Co-design       | 算法与实现协同设计               |
| P15 | Data-Centric Optimization          | 数据选择和生成为主要杠杆         |

**输出**：主要范式（1个）+ 次要范式（0-2个）+ 置信度 + 分类理由

## 七、技术决策

### 7.1 LLM Prompt管理

**现状**：

- 支持多版本prompt（v1/v2/v3）
- 每个生成器（Summary/Prior Works/Pattern）都可独立切换prompt，并可单独重跑
- Prompt以`.py`文件形式存储，便于版本管理

**原则**：

- 持续优化prompt质量（对标ResearchStudio的prompt工程）
- 保留所有历史版本（便于回滚和对比）
- 记录prompt版本与生成结果的对应关系（在metadata中）

---

### 7.2 人工审查的必要性

**为什么需要人工审查**：

- LLM在提取前作时存在幻觉风险（论文标题、arXiv ID可能错误）
- 论文元数据（作者、会议、年份）需要确认
- Summary中的关键结论需要校对

**审查流程**：

1. 系统生成Summary、Prior Works、Pattern
2. 用户打开Review UI，检查关键信息
3. 在 `manifest.json` 的 `paper` 字段中修正论文元信息，并标记 `reviewed: true`
4. 仅有 `manifest.paper.reviewed: true` 的论文才能入库和发布

**预期**：随着LLM能力提升和prompt优化，人工审查时间会逐渐减少（但永不消失）。

---

### 7.3 增量更新与幂等性

**设计原则**：

- Assets构建总是deterministic（重跑结果相同）
- LLM生成可能变化（prompt更新、模型更新），但不回溯覆盖已审查版本
- 图谱更新通过事件日志实现，支持幂等重放（相同事件可安全重新应用）

**实现**：

- `--overwrite` 标志：仅当用户明确要求时才覆盖已有产物
- `graph_state/` + `graph_updates.jsonl`：快照+事件日志，支持时间旅行

## 十、前端原型设计（基于alphaxiv.org的交互模式）

### 10.1 为什么需要Web界面而不仅仅是CLI

**现状问题**：

- CLI命令需要记忆和打字，降低发现效率
- 无法实时查看论文卡片、图片预览、Summary摘要
- 审查Summary/Prior Works时，需要手工编辑JSON，体验糟糕
- 知识图谱只存在数据库中，用户无法可视化探索

**Web界面的价值**：

- 📱 **论文发现**：列表/搜索/推荐一体化，支持快速浏览
- 👁️ **即时预览**：悬停卡片显示Summary摘要，点击展开完整内容
- ✏️ **人工审查**：所见即所得编辑Summary/Prior Works，一键保存

## 十一、Web原型 v1 技术方案（工程实现）

> 本节把第十节的交互设想收敛为**第一个可落地的版本**：明确包结构、数据契约、日志、依赖隔离和里程碑，目标是让工程实现过程可维护、可回滚、不越界过度设计。

**当前实现状态（2026-07-14）**：Web v1 已完整落地 M1-M7。后端 `paper_wiki.web` FastAPI 入口层新增 `recommendations`/`search` router，并在 `papers` router 上补齐 `POST /api/papers/fetch`、`POST /api/papers/batch-ingest`；`JobManager`/`JobResponse` 扩展了 `result` 字段，使 fetch/recommend-refresh/batch-ingest 这类"结果里带新 slug"的异步任务也能通过既有轮询模型消费。前端新增 `TodayFeed`（首页，读 `/api/recommendations/today`，支持勾选批量生成）与 `SearchAndAdd`（关键词检索/按 arXiv ID 拉取，两个入口都能把候选"加入待筛选"汇入 TodayFeed 的候选池）；`ReviewQueue` 未新建组件，而是复用既有 Dashboard+PaperDetail 分栏视图并固定 `reviewed=false` 过滤，导航栏改为"今日推荐 / 检索添加 / 全部论文 / 待审查(N)"四个入口。批量生成的候选按"是否已有本地 slug"分流：已知 slug 的走 `IngestPipeline.run`；仅有 arXiv ID 的复用 `discovery.search.fetch(and_ingest=True)`（内部仍是同一个 `IngestPipeline.run`），两条路径共享同一个 `JobManager`，不新增队列框架。附带修复了 `daily_candidates()`（arXiv 每日 RSS）解析出的 `arxiv_id` 未剥离 `oai:arXiv.org:` 前缀的问题——该前缀是 RSS Atom `<id>` 特有格式，此前会导致今日推荐候选无法被正确 fetch。图谱可视化、上传、多平台发布和鉴权仍按 v1 裁剪范围保持未实现。

**后续体验优化（2026-07-15）**：M7上线后基于真实使用反馈做了一轮打磨，详见 [Paper-Wiki 推荐Feed与审查体验优化-技术方案.md](<./Paper-Wiki%20推荐Feed与审查体验优化-技术方案.md>)。其中一处涉及数据模型变化需要在此说明：`manifest.json: paper.reviewed` 不再由客户端直接赋值，而是拆成 `meta_reviewed`（中栏元信息+摘要审查）和 `prior_works_reviewed`（右栏前作审查）两个开关，`reviewed = meta_reviewed and prior_works_reviewed` 由服务端派生；`PaperAssetMeta` 加了一个隐式迁移校验器，兼容升级前只有单一 `reviewed` 字段的旧 manifest。TodayFeed 卡片新增 LLM 生成的一句话候选摘要（`RankedCandidate.display_summary`，仅对 Top K 生成，不违反"发现层不调用LLM"的边界——生成逻辑放在 Web 层的 `web/services/candidate_summary.py`）；异步任务（刷新推荐/批量生成/单篇ingest）现在会通过 `JobResponse.progress` 展示翻译过的中文进度提示，而不是干等一个转圈。

### 11.0 设计原则

- **单一数据源**：v1不引入新数据库，`artifacts/` 目录 + `manifest.json` 仍是唯一事实来源，避免文件系统与DB双写不一致。
- **复用而非重写**：Web层只做编排（orchestration），业务逻辑继续留在 `ingestion` / `assets` / `graph` / `publishing` 包里，不重复实现。
- **依赖方向单向**：`web` 依赖 `core/assets/ingestion/graph/publishing`；反向依赖禁止。`cli` 和 `web` 是两个平行的入口层，都只调用同一套 service；CLI 只允许为 `paper-wiki web` 启动命令延迟导入 Web app/logging，不在业务命令中耦合 Web 路由或 schema。
- **垂直切片交付**：每个里程碑都是可独立验收、可独立上线的完整闭环，不做"半成品"堆叠。

**依赖规则**（建议用简单脚本或 `import-linter` 在CI里强制，而不是靠口头约定）：

- `paper_wiki.web.*` 和 `paper_wiki.discovery.*` 都可以 import `paper_wiki.{core,assets,ingestion,graph,publishing}`

### 11.2 数据契约：为什么v1不引入数据库

现状：`manifest.json` 已包含 `paper.reviewed` 字段（见 `assets/models.py: PaperAssetMeta`），并且已经是 Pydantic 校验过的结构化数据。v1 Web 直接读写这些文件即可满足"发现→审查→发布"闭环，不需要额外引入 PostgreSQL。

**优点**：

- 避免文件系统与DB的双写一致性问题——这是当前架构的核心不变式，一旦打破会让 CLI 和 Web 的状态互相"打架"
- CLI 和 Web 天然共享同一份状态，不会出现"Web上显示已审查，CLI里还没标记"的撕裂
- 复杂度可控，符合MVP优先级，不为尚未验证的规模问题预先设计

**代价与应对**：

- 论文列表需遍历目录+解析每个 `manifest.json`；论文量在数百篇内可接受。超过后可引入 SQLite 做**只读索引缓存**（不是数据源，可随时从 `artifacts/` 重建），而不是替换文件系统本身
- 并发写入：v1只有单用户本地场景，用乐观锁（比较请求携带的 `expected_updated_at` 与 `manifest.updated_at`）+ `filelock` 防止两个请求同时写同一篇论文，冲突返回 `409 Conflict`

### 11.3 核心服务设计

**PaperRepository**（`web/services/paper_repository.py`）

- `list_papers() -> list[PaperListItem]`：遍历 `artifacts/*/manifest.json`
- `get_paper(slug) -> PaperDetail`：组合 `AssetsReader.load()` + `summary.md` + `prior_works.json` + `sci_pattern.json`
- `update_meta(slug, patch, *, expected_updated_at)` / `update_summary(...)` / `update_prior_works(...)`：写回前先用 `filelock` 加锁，并校验 `expected_updated_at`
- 复用现有的 slug 路径穿越校验（建议把 `AssetsReader._validate_slug` 提升为 `paper_wiki.core.validation.validate_slug` 供 web/assets 共用，而不是在 web 层重新实现一遍）

**JobManager**（`web/services/job_manager.py`）

- v1：进程内 `dict[str, JobRecord]` + `ThreadPoolExecutor`（`IngestPipeline` 调用是同步阻塞的LLM请求，不能直接 `await`）
- 每个 Job 记录：`job_id, slug, target, status(pending/running/succeeded/failed), started_at, finished_at, error`
- **明确不做**崩溃恢复持久化队列：进程重启后运行中任务丢失，前端提示"请重新触发"——这是刻意的取舍，为单用户本地工具引入 Celery/Redis 属于过度设计
- `JobManager` 以抽象接口暴露给 router

### 11.4 日志与可观测性

延续 `core/logging.py` 的思路（stdlib logging，不为此引入新依赖）：

1. `configure_web_logging(verbose)`：在 `configure_logging` 基础上，给 `uvicorn.access` / `uvicorn.error` / `paper_wiki.web` logger 挂同样的 formatter，保证 CLI 和 Web 的日志格式一致，排查问题时不用切换心智模型
2. `web/middleware.py: RequestIDMiddleware`：用 `contextvars` 存 `request_id`，通过 `logging.Filter` 注入 Web logger；请求结束打一条 INFO：`method path status duration_ms`
3. 后台 Job 复用 pipeline 里已有的 `logger.info("步骤 %d/%d: ...")`，Web层不重复造日志；v1 的 `JobRecord` 暴露 `job_id/status/error` 用于前端轮询和人工排查，后续可再补 LoggerAdapter 绑定 job_id
4. 错误处理统一收口在 `error_handlers.py`：

- 业务异常（`FileExistsError`、`PublishingError`、校验失败）→ WARNING + 4xx
- 未预期异常 → ERROR + 堆栈 + 500。区别于CLI"verbose才打堆栈"的习惯，Web端默认总是记录堆栈（没有人盯着终端交互式排查）

| POST   | `/api/recommendations/refresh` | 异步触发重新计算今日推荐，返回`job_id`，成功后 `job.result` 即最新快照                                                                                                                                   | `paper_wiki.discovery.recommend` |
| GET    | `/api/search`                  | 按主题检索arXiv候选（`?q=&start_year=&end_year=&max_results=`）                                                                                                                                            | `paper_wiki.discovery.search`    |
| POST   | `/api/papers/fetch`            | 异步按arXiv ID拉取PDF/LaTeX到`raw/{slug}/`，返回`job_id`；`job.result` 含解析出的 slug                                                                                                                 | `paper_wiki.discovery.fetch`     |
| POST   | `/api/papers/batch-ingest`     | 批量触发多篇论文生成三件套，body为`{items:[{arxiv_id}或{slug}], overwrite}`，返回`job_id` 列表（按 slug/arxiv_id 分流到 `IngestPipeline.run` 或 `discovery.fetch(and_ingest=True)`，不新增队列框架） | `IngestPipeline.run` × N        |

`JobRecord`/`JobResponse` 新增 `result: Any \| None` 字段（任务成功后写入 `jsonable_encoder` 序列化的返回值），使 fetch/refresh/batch-ingest 这类"结果里带新数据（新 slug、新快照）"的异步任务，前端轮询到终态后直接读 `job.result` 即可，不必再新增查询接口。

**v1明确不做**（写进文档，避免范围蔓延）：

- 论文上传/PDF解析入口（继续用 `raw/{slug}/` + CLI `ingest`）
- 按主题的多源检索（Semantic Scholar/OpenAlex等）——v1只做arXiv单源+按ID拉取，多源检索是v2
- 图谱查询与可视化接口
- 多平台发布
- 鉴权（v1假设本地单用户；上线到多用户前必须补上）

### 11.6 前端 v1

技术选型收敛到最小可用集：**Vite + React + TypeScript + TailwindCSS**（不用Next.js，本地单用户工具不需要SSR/路由预渲染的复杂度）。

对应"发现"和"审查发布"两个核心场景，10.3流程4的图谱可视化推迟到v2：

- `TodayFeed`：✅ 已实现（M7，首页/默认视图），读 `GET /api/recommendations/today`；卡片支持勾选多选 + 顶部"批量生成"，内部按候选是否已带本地 `slug`（来自 `SearchAndAdd` 的 fetch-only 结果）拆成 `{slug}` 或 `{arxiv_id}` 两类调用 `POST /api/papers/batch-ingest`；"刷新推荐"按钮调用 `POST /api/recommendations/refresh` 并轮询job
- `SearchAndAdd`：✅ 已实现（M7），两个 Tab：关键词检索（读 `GET /api/search`，勾选后"加入待筛选"把候选提升到 `App` 级别的 `staged` 状态，供 `TodayFeed` 消费）与按arXiv ID添加（写 `POST /api/papers/fetch`，可选 `and_ingest`；不立即生成时同样进入 `staged` 池）
- `Dashboard` + `PaperDetail`：✅ 已实现，论文卡片列表（读 `GET /api/papers`）+ 详情/编辑/发布；导航栏的"全部论文"与"待审查 (N)"两个入口复用同一对组件，分别以 `reviewed=undefined`/`reviewed=false` 驱动列表过滤——**没有**新建独立的 `ReviewQueue` 组件，"待审查队列"就是这对组件加上固定过滤条件的一个视图

前端只允许通过 `src/api/client.ts` 访问后端（唯一出口），组件内不得散落 fetch 调用——以后要处理乐观锁冲突提示、换成 openapi-generated client，只需改这一个文件。`client.ts` 新增的 `pollJob()` 是 fetch/refresh/batch-ingest/单篇ingest 共用的轮询封装，替换了此前 `PaperDetail` 里手写的 `setInterval`。
